# Product Requirements Document: Library Events Producer

## 1. Project Overview

**Application Name:** Library Events Producer v2  
**Purpose:** REST API service to publish library events (ADD, UPDATE) to Apache Kafka for downstream consumers  
**Technology Stack:** Spring Boot 4.1.0, Java 25, Gradle, Kafka  
**Status:** In Development

---

## 2. Goals & Success Metrics

- Expose reliable REST endpoints for publishing library events to Kafka
- Publish events with 99.9% success rate (producer acknowledgments)
- Median end-to-end publish latency: <200ms
- All integration tests passing in CI/CD pipeline
- Clear, maintainable code with comprehensive test coverage

## 5. Data Model

### LibraryEvent Structure

```json
{
  "libraryEventId": 123,
  "eventType": "ADD",
  "book": {
    "bookId": 1,
    "bookName": "Clean Code",
    "bookAuthor": "Robert C. Martin"
  }
}
```

### Field Definitions

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| libraryEventId | Integer | Yes | Generated by upstream; uniquely identifies the event |
| eventType | Enum | Yes | Values: `ADD`, `UPDATE` |
| book.bookId | Integer | Yes | Must be positive |
| book.bookName | String | Yes | Non-empty, max 255 characters |
| book.bookAuthor | String | Yes | Non-empty, max 255 characters |

### Event Types

- **ADD:** Published when a new book is added to the library
- **UPDATE:** Published when an existing book's details are updated

## 6. API Endpoints

### Base URL
```
/v1/library-events
```

### 6.1 POST /v1/library-events

**Description:** Publish a new library event with event type ADD

**Request Body:**
```json
{
  "libraryEventId": 123,
  "eventType": "ADD",
  "book": {
    "bookId": 1,
    "bookName": "Clean Code",
    "bookAuthor": "Robert C. Martin"
  }
}
```

**Success Response:**
- Status: `201 Created`
- Headers: `Location: /v1/library-events/123`
- Body: Event JSON (echo back with confirmation)

**Error Responses:**
- `400 Bad Request`: Validation failure (missing fields, invalid types, invalid eventType)
- `503 Service Unavailable`: Kafka publish failure (after retries exhausted)

**Validation Rules:**
- libraryEventId: required, positive integer
- eventType: must be "ADD"
- book.bookId: required, positive integer
- book.bookName: required, non-empty string, ≤255 chars
- book.bookAuthor: required, non-empty string, ≤255 chars

---

### 6.2 PUT /v1/library-events

**Description:** Publish an update to an existing library event with event type UPDATE

**Request Body:**
```json
{
  "libraryEventId": 123,
  "eventType": "UPDATE",
  "book": {
    "bookId": 1,
    "bookName": "Clean Code (Updated Edition)",
    "bookAuthor": "Robert C. Martin"
  }
}
```

**Success Response:**
- Status: `200 OK`
- Body: Event JSON

**Error Responses:**
- `400 Bad Request`: Validation failure or missing libraryEventId
- `503 Service Unavailable`: Kafka publish failure

**Validation Rules:**
- libraryEventId: required, positive integer
- eventType: must be "UPDATE"
- book.bookId: required, positive integer
- book.bookName: required, non-empty string, ≤255 chars
- book.bookAuthor: required, non-empty string, ≤255 chars

## 7. Kafka Integration

### Producer Configuration

| Property | Value | Rationale |
|----------|-------|-----------|
| Topic | `library-events` | Configurable via properties |
| Message Key | String (libraryEventId) | Ensures ordering of events per library event |
| Message Value | JSON payload | Structured event data |
| acks | all | Ensures durability (all replicas acknowledge) |
| retries | 3 | Automatic retry on transient failures |
| enable.idempotence | true | Prevents duplicate messages |
| linger.ms | 10 | Small batching window for throughput |

### Kafka Message Format

**Key:** String representation of libraryEventId  
**Value:** JSON serialized LibraryEvent object

**Example Message:**
```
Key: "123"
Value: {"libraryEventId":123,"eventType":"ADD","book":{"bookId":1,"bookName":"Clean Code","bookAuthor":"Robert C. Martin"}}
```

### Delivery Semantics

- **Guarantee:** At-least-once (idempotent producer prevents duplicates from retries)
- **Error Handling:**
  - Transient Kafka errors: Retry with exponential backoff
  - Persistent failures: Log error and return HTTP 503
  - No dead-letter queue required for MVP (can be added in Phase 2)

## 8. Validation Rules

### Input Validation

All incoming requests must validate:

1. **Request Body:** Valid JSON structure
2. **Field Types:** Correct data types (integer, string, enum)
3. **Field Values:**
   - libraryEventId: positive integer
   - eventType: one of [ADD, UPDATE]
   - bookId: positive integer
   - bookName: non-empty string, ≤255 characters
   - bookAuthor: non-empty string, ≤255 characters
4. **Business Rules:**
   - No existence checking required (no persistence layer)

### Error Response Format

```json
{
  "timestamp": "2026-07-04T00:47:28Z",
  "status": 400,
  "error": "Bad Request",
  "message": "Validation failed",
  "details": [
    "bookName must not be empty",
    "bookId must be positive"
  ]
}
```

## 9. Error Handling & Resilience

### Synchronous Publishing
- REST endpoint waits for producer to complete before returning response
- Success only returned after Kafka producer ACK

### Failure Scenarios

| Scenario | Handling | Response |
|----------|----------|----------|
| Invalid payload | Validation exception | 400 Bad Request |
| Kafka broker unavailable | Retry with backoff; if exhausted, fail | 503 Service Unavailable |
| Network timeout | Retry with exponential backoff | 503 if retries fail |
| Invalid message format | Producer exception logged | 503 Service Unavailable |

### Logging
- All errors logged with correlation ID (from request context)
- Stack traces logged at ERROR level for debugging

## 12. Testing Strategy

### Unit Tests
- Controller layer: Validate request/response mapping, error handling
- Service layer: Kafka producer calls, business logic
- Validation: Field validation logic
- Mocks: KafkaTemplate mocked; no real broker needed

### Integration Tests
- **EmbeddedKafka:** Real Kafka broker embedded in tests
- Assertions: Verify messages published to topic with correct key/value
- Event payload correctness
- End-to-end flow (POST/PUT → Kafka message)

### Test Scenarios

| Test Case | Expected Result |
|-----------|-----------------|
| POST with valid ADD event | 201 Created, message in topic |
| PUT with valid UPDATE event | 200 OK, message in topic |
| POST with missing field | 400 Bad Request |
| POST with invalid eventType | 400 Bad Request |
| POST with non-positive bookId | 400 Bad Request |
| Kafka failure on publish | 503 Service Unavailable |
| Idempotency: duplicate publish | Single message in topic (after retries) |
